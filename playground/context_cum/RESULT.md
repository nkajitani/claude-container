Claude API におけるコンテキスト消費の構造的分析

今回の検証を通じて明らかになった、LLMのトークン消費に関する「三層構造」の分析結果です。

1. トークン消費の三要素

APIの利用において、以下の3つのポイントでコンテキスト（トークン）が発生しています。

A. 蓄積型インプット (Input Tokens)

内容: [システムプロンプト] + [過去の全履歴] + [今回のプロンプト]

特性: 会話が続く限り、過去のアウトプットが次回のインプットに化けるため、指数関数的に増大します。

今回の結果: Turn 1 (41) → Turn 2 (231) → Turn 3 (392) と、過去のやり取りを丸ごと飲み込んでいます。

B. 生成型アウトプット (Output Tokens)

内容: AIがその場で新しく書き出した言葉。

特性: AIが「沈黙」を選ばない限り、最低限の枕詞や確認の言葉が発生します。

今回の謎: 全く同じ質問でもアウトプットが発生するのは、AIが「過去の回答がある」という前提で「新しい返答」を生成しようとする推論の結果です。

C. 思考のオーバーヘッド

内容: インプットからアウトプットを導き出すための計算リソース。

特性: インプット量が増えるほど、AIの「注目すべき点（Attention）」が分散し、レスポンス速度の低下や精度の揺らぎに繋がります。

2. なぜ「同じ質問」でコストがかかるのか？

人間であれば「さっきの回答を見て」で済みますが、API（ステートレスなLLM）にとっては：

「過去の回答」も今回新しく読み込ませたカンニングペーパーの一部。

そのペーパーを読み直す作業（インプット課金）がまず発生する。

その上で、再度「回答という作業」をゼロから行い、文字を書き出す（アウトプット課金）ため。

3. 実務へのフィードバック

この構造を理解した上での最適化戦略：

キャッシュの活用: CLAUDE.md のような固定情報は、システムのキャッシュ機能に乗りやすく、再計算コストを下げられる。

履歴のトリミング: 不要になった過去の assistant 発言を削除、または要約してから次に渡すことで、インプットの肥大化を食い止める。

明確な停止指示: 「同じ内容なら『同上』とだけ返して」と指示することで、アウトプット・トークン（生成コスト）を最小化できる。
